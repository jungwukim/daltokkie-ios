# 달토끼 iOS — 개발 프로세스 (PDCA + TDD)

> 모든 기능 개발은 이 프로세스를 따른다.
> 각 단계의 산출물은 `docs/features/{feature-name}/` 하위에 저장한다.

## 워크플로우

```
[요청] → Plan → Design → TDD(Red→Green→Refactor) → Check → Act
         ↓       ↓              ↓                     ↓      ↓
      PLAN.md  DESIGN.md    테스트→구현→리팩터       검증   WORKLOG 기록
```

## 단계별 상세

### 1. Plan (계획)

**산출물**: `docs/features/{name}/PLAN.md`

작성 내용:
- 요청 원문 (사용자가 말한 그대로)
- 변경 범위 (영향받는 파일 목록)
- 하지 않을 것 (범위 밖 명시)
- 리스크/질문 사항

```markdown
# Feature: {이름}

## 요청
> 사용자 요청 원문

## 변경 범위
- [ ] 파일1: 변경 내용
- [ ] 파일2: 변경 내용

## 하지 않을 것
- 범위 밖 사항 명시

## 리스크/질문
- 확인 필요 사항
```

**게이트**: 사용자가 Plan을 확인하고 승인해야 다음 단계로 진행

### 2. Design (설계)

**산출물**: `docs/features/{name}/DESIGN.md`

작성 내용:
- 기술 설계 (구조, 타입, 인터페이스)
- 근거 (왜 이 방식인지 — 공식 문서, 기존 패턴, 기술 제약)
- 변경 전/후 비교

```markdown
# Design: {이름}

## 설계

### 변경 전
현재 코드/구조 설명

### 변경 후
제안하는 코드/구조 설명

## 근거
- 근거 1: (출처)
- 근거 2: (출처)

## 영향 분석
- 기존 기능 영향 여부
```

**게이트**: 사용자가 Design을 확인하고 승인해야 구현 시작

### 3. Do — TDD (구현)

**순서**: Red → Green → Refactor

| 단계 | 내용 |
|------|------|
| **Red** | 실패하는 테스트를 먼저 작성. 테스트가 실패하는지 확인 |
| **Green** | 테스트를 통과하는 최소한의 코드 작성. 테스트 통과 확인 |
| **Refactor** | 테스트가 통과하는 상태에서 코드 정리. 테스트 재확인 |

**규칙**:
- Engine 레이어: `cd Engine && swift test`로 검증
- App 레이어: `xcodebuild test` 또는 시뮬레이터 확인
- 테스트 없이 구현하지 않음 (테스트 불가능한 UI 변경은 시뮬레이터 확인으로 대체)
- 요청 범위만 구현 — 추가 변경은 사용자에게 질문

### 4. Check (검증)

검증 항목:
- [ ] Plan의 변경 범위가 모두 반영되었는지
- [ ] 테스트 전부 통과하는지
- [ ] 요청하지 않은 변경이 없는지
- [ ] 기존 기능에 영향이 없는지

### 5. Act (기록)

작업 완료 후:
- `docs/WORKLOG.md`에 히스토리 추가
- `docs/DECISIONS.md`에 설계 결정 기록 (해당되는 경우)
- 남은 작업 목록 업데이트

## 디렉토리 구조

```
docs/
├── WORKLOG.md          ← 앵커 문서 (히스토리, 상태, 남은 작업)
├── DECISIONS.md        ← 설계 결정 기록
├── PROCESS.md          ← 이 문서 (프로세스 정의)
└── features/           ← 기능별 PDCA 산출물
    └── {feature-name}/
        ├── PLAN.md
        └── DESIGN.md
```

## 예시: "탭바 높이 조정" 기능이었다면

```
docs/features/tabbar-height-fix/
├── PLAN.md    → 요청: offset 조정만, 아이콘/라벨 변경 안 함
└── DESIGN.md  → .offset(y: -20) → .offset(y: 0), 근거: 사용자 요청
```

## 테스트 불가능한 변경의 경우

UI 변경처럼 자동 테스트가 어려운 경우:
1. Design 단계에서 변경 전/후를 명확히 기술
2. 구현 후 시뮬레이터에서 확인
3. 사용자에게 결과 보고 후 승인 요청

## 프로세스 적용 기준

| 변경 규모 | 프로세스 |
|----------|---------|
| 한 줄 수정, 오타 | Plan/Design 생략, WORKLOG에 기록만 |
| 단일 파일 변경 | Plan 간략 작성, Design 생략 가능, WORKLOG 기록 |
| 다중 파일 / 새 기능 | 전체 PDCA 프로세스 적용 |
| 아키텍처 변경 | 전체 PDCA + DECISIONS 기록 필수 |
